6//! This crate contains Winterfell STARK prover and verifier.
7//!
8//! A STARK is a novel proof-of-computation scheme to create efficiently verifiable proofs of the
9//! correct execution of a computation. The scheme was developed by Eli Ben-Sasson, Michael Riabzev
10//! et al. at Technion - Israel Institute of Technology. STARKs do not require an initial trusted
11//! setup, and rely on very few cryptographic assumptions. See [references](#references) for more
12//! info.
13//!
14//! ## Proof generation
15//! To generate a proof that a computation was executed correctly, you'll need to do the
16//! following:
17//!
18//! 1. Define an *algebraic intermediate representation* (AIR) for your computation. This can
19//!    be done by implementing [Air] trait.
20//! 2. Define an execution trace for your computation. This can be done by implementing [Trace]
21//!    trait. Alternatively, you can use the [TraceTable] struct which already implements [Trace]
22//!    trait in cases when this generic implementation works for your use case.
23//! 3. Execute your computation and record its execution trace.
24//! 4. Define your prover by implementing [Prover] trait. Then execute [Prover::prove()] function
25//!    passing the trace generated in the previous step into it as a parameter. The function will
26//!    return an instance of [Proof].
27//!
28//! This `Proof` can be serialized and sent to a STARK verifier for verification. The size
29//! of proof depends on the specifics of a given computation, but for most computations it should
30//! be in the range between 15 KB (for very small computations) and 300 KB (for very large
31//! computations).
32//!
33//! Proof generation time is also highly dependent on the specifics of a given computation, but
34//! also depends on the capabilities of the machine used to generate the proofs (i.e. on the number
35//! of CPU cores and memory bandwidth).
36//!
37//! When the crate is compiled with the `concurrent` feature enabled, proof generation will be
38//! performed in multiple threads (usually, as many threads as there are logical cores on the
39//! machine). The number of threads can be configured via `RAYON_NUM_THREADS` environment
40//! variable.
41//!
42//! ## Prof verification
43//! To verify a [Proof] generated as described in the previous sections, you'll need to
44//! do the following:
45//!
46//! 1. Define an *algebraic intermediate representation* (AIR) for you computation. This AIR
47//!    must be the same as the one used during the proof generation process.
48//! 2. Execute [verify()] function and supply the AIR of your computation together with the
49//!    [Proof] and related public inputs as parameters.
50//!
51//! Proof verification is extremely fast and is nearly independent of the complexity of the
52//! computation being verified. In the vast majority of cases, proofs can be verified in 3 - 5 ms
53//! on a modern mid-range laptop CPU (using a single core).
54//!
55//! There is one exception, however: if a computation requires a lot of `sequence` assertions
56//! (see [Assertion] for more info), the verification time will grow linearly in the number of
57//! asserted values. But for the impact to be noticeable, the number of asserted values would
58//! need to be in tens of thousands. And even for hundreds of thousands of assorted values, the
59//! verification time should not exceed 50 ms.
60//!
61//! # Examples
62//! The best way to understand the STARK proof generation and verification process is to go
63//! through a trivial example from start to finish. First, we'll need to pick a computation for
64//! which we'll be generating and verifying STARK proofs. To keep things simple, we'll use the
65//! following:
66//!
67//! ```no_run
68//! use winterfell::math::{fields::f128::BaseElement, FieldElement};
69//!
70//! fn do_work(start: BaseElement, n: usize) -> BaseElement {
71//!    let mut result = start;
72//!    for _ in 1..n {
73//!        result = result.exp(3) + BaseElement::new(42);
74//!    }
75//!    result
76//! }
77//! ```
78//!
79//! This computation starts with an element in a finite field and then, for the specified number
80//! of steps, cubes the element, and adds value `42` to it.
81//!
82//! Suppose, we run this computation for a million steps and get some result. Using STARKs we can
83//! Prove that we did the work correctly without requiring any verifying party to re-execute the
84//! computation. Here is how to do it:
85//!
86//! First, we need to define an *execution trace* for our computation. This trace should capture
87//! the state of the computation at every step of its execution. In our case, the trace is just a
88//! single column of intermediate values after each execution of the loop. For example, if we start
89//! with value `3` and run the computation for 1,048,576 (same as 2<sup>20</sup>) steps, the
90//! execution trace will look like this:
91//!
92//! | Step      | State  |
93//! | :-------: | :----- |
94//! | 0         | 3      |
95//! | 1         | 69     |
96//! | 2         | 328551 |
97//! | 3         | 35465687262668193 |
98//! | 4         | 237280320818395402166933071684267763523 |
99//! | ...       |
100//! | 1,048,575 | 247770943907079986105389697876176586605 |
101//!
102//! To record the trace, we'll use the [TraceTable] struct. The function below is just a
103//! modified version of the `do_work()` function which records every intermediate state of the
104//! computation in the [TraceTable] struct:
105//!
106//! ```no_run
107//! use winterfell::{
108//!     math::{fields::f128::BaseElement, FieldElement},
109//!     TraceTable,
110//! };
111//!
112//! pub fn build_do_work_trace(start: BaseElement, n: usize) -> TraceTable<BaseElement> {
113//!     // Instantiate the trace with a given width and length; this will allocate all
114//!     // required memory for the trace
115//!     let trace_width = 1;
116//!     let mut trace = TraceTable::new(trace_width, n);
117//!
118//!     // Fill the trace with data; the first closure initializes the first state of the
119//!     // computation; the second closure computes the next state of the computation based
120//!     // on its current state.
121//!     trace.fill(
122//!         |state| {
123//!             state[0] = start;
124//!         },
125//!         |_, state| {
126//!             state[0] = state[0].exp(3u32.into()) + BaseElement::new(42);
127//!         },
128//!     );
129//!
130//!     trace
131//! }
132//! ```
133//!
134//! Next, we need to define *algebraic intermediate representation* (AIR) for our computation.
135//! This process is usually called *arithmetization*. We do this by implementing the [Air] trait.
136//! At the high level, the code below does three things:
137//!
138//! 1. Defines what the public inputs for our computation should look like. These inputs are
139//!    called "public" because they must be known to both, the prover and the verifier.
140//! 2. Defines a transition function with a single transition constraint. This transition
141//!    constraint must evaluate to zero for all valid state transitions, and to non-zero for any
142//!    invalid state transition. The degree of this constraint is 3 (see more about constraint
143//!    degrees "Constraint degrees" section of [Air] trait documentation).
144//! 3. Define two assertions against an execution trace of our computation. These assertions tie
145//!    a specific set of public inputs to a specific execution trace (see more about assertions
146//!    "Trace assertions" section of [Air] trait documentation).
147//!
148//! Here is the actual code:
149//!
150//! ```no_run
151//! use winterfell::{
152//!     math::{fields::f128::BaseElement, FieldElement, ToElements},
153//!     Air, AirContext, Assertion, GkrVerifier, EvaluationFrame,
154//!     ProofOptions, TraceInfo, TransitionConstraintDegree,
155//!     crypto::{hashers::Blake3_256, DefaultRandomCoin, MerkleTree},
156//! };
157//!
158//! // Public inputs for our computation will consist of the starting value and the end result.
159//! pub struct PublicInputs {
160//!     start: BaseElement,
161//!     result: BaseElement,
162//! }
163//!
164//! // We need to describe how public inputs can be converted to field elements.
165//! impl ToElements<BaseElement> for PublicInputs {
166//!     fn to_elements(&self) -> Vec<BaseElement> {
167//!         vec![self.start, self.result]
168//!     }
169//! }
170//!
171//! // For a specific instance of our computation, we'll keep track of the public inputs and
172//! // the computation's context which we'll build in the constructor. The context is used
173//! // internally by the Winterfell prover/verifier when interpreting this AIR.
174//! pub struct WorkAir {
175//!     context: AirContext<BaseElement>,
176//!     start: BaseElement,
177//!     result: BaseElement,
178//! }
179//!
180//! impl Air for WorkAir {
181//!     // We'll specify which finite field to use for our computation, and also how
182//!     // the public inputs must look like.
183//!     type BaseField = BaseElement;
184//!     type PublicInputs = PublicInputs;
185//!     type GkrProof = ();
186//!     type GkrVerifier = ();
187//!
188//!     // Here, we'll construct a new instance of our computation which is defined by 3
189//!     // parameters: starting value, number of steps, and the end result. Another way to
190//!     // think about it is that an instance of our computation is a specific invocation of
191//!     // the do_work() function.
192//!     fn new(trace_info: TraceInfo, pub_inputs: PublicInputs, options: ProofOptions) -> Self {
193//!         // our execution trace should have only one column.
194//!         assert_eq!(1, trace_info.width());
195//!
196//!         // Our computation requires a single transition constraint. The constraint itself
197//!         // is defined in the evaluate_transition() method below, but here we need to specify
198//!         // the expected degree of the constraint. If the expected and actual degrees of the
199//!         // constraints don't match, an error will be thrown in the debug mode, but in release
200//!         // mode, an invalid proof will be generated which will not be accepted by any verifier.
201//!         let degrees = vec![TransitionConstraintDegree::new(3)];
202//!
203//!         // We also need to specify the exact number of assertions we will place against the
204//!         // execution trace. This number must be the same as the number of items in a vector
205//!         // returned from the get_assertions() method below.
206//!         let num_assertions = 2;
207//!
208//!         WorkAir {
209//!             context: AirContext::new(trace_info, degrees, num_assertions, options),
210//!             start: pub_inputs.start,
211//!             result: pub_inputs.result,
212//!         }
213//!     }
214//!
215//!     // In this method we'll define our transition constraints; a computation is considered to
216//!     // be valid, if for all valid state transitions, transition constraints evaluate to all
217//!     // zeros, and for any invalid transition, at least one constraint evaluates to a non-zero
218//!     // value. The `frame` parameter will contain current and next states of the computation.
219//!     fn evaluate_transition<E: FieldElement + From<Self::BaseField>>(
220//!         &self,
221//!         frame: &EvaluationFrame<E>,
222//!         _periodic_values: &[E],
223//!         result: &mut [E],
224//!     ) {
225//!         // First, we'll read the current state, and use it to compute the expected next state
226//!         let current_state = frame.current()[0];
227//!         let next_state = current_state.exp(3u32.into()) + E::from(42u32);
228//!
229//!         // Then, we'll subtract the expected next state from the actual next state; this will
230//!         // evaluate to zero if and only if the expected and actual states are the same.
231//!         result[0] = frame.next()[0] - next_state;
232//!     }
233//!
234//!     // Here, we'll define a set of assertions about the execution trace which must be
235//!     // satisfied for the computation to be valid. Essentially, this ties computation's
236//!     // execution trace to the public inputs.
237//!     fn get_assertions(&self) -> Vec<Assertion<Self::BaseField>> {
238//!         // for our computation to be valid, value in column 0 at step 0 must be equal to the
239//!         // starting value, and at the last step it must be equal to the result.
240//!         let last_step = self.trace_length() - 1;
241//!         vec![
242//!             Assertion::single(0, 0, self.start),
243//!             Assertion::single(0, last_step, self.result),
244//!         ]
245//!     }
246//!
247//!     // This is just boilerplate which is used by the Winterfell prover/verifier to retrieve
248//!     // the context of the computation.
249//!     fn context(&self) -> &AirContext<Self::BaseField> {
250//!         &self.context
251//!     }
252//! }
253//! ```

581//!     32, // number of queries
582//!     8,  // blowup factor
583//!     0,  // grinding factor
584//!     FieldExtension::None,
585//!     8,  // FRI folding factor
586//!     31, // FRI max remainder polynomial degree
587//! );
588//!
589//! // Instantiate the prover and generate the proof.
590//! let prover = WorkProver::new(options);
591//! let proof = prover.prove(trace).unwrap();
592//!
593//! // The verifier will accept proofs with parameters which guarantee 95 bits or more of
594//! // conjectured security
595//! let min_opts = winterfell::AcceptableOptions::MinConjecturedSecurity(95);
596//!
597//! // Verify the proof. The number of steps and options are encoded in the proof itself,
598//! // so we don't need to pass them explicitly to the verifier.
599//! let pub_inputs = PublicInputs { start, result };
600//! assert!(winterfell::verify::<WorkAir,
601//!                              Blake3_256<BaseElement>,
602//!                              DefaultRandomCoin<Blake3_256<BaseElement>>,
603//!                              MerkleTree<Blake3_256<BaseElement>>
604//!                             >(proof, pub_inputs, &min_opts).is_ok());
605//! ```
